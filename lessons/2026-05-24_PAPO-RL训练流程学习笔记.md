## 一、训练循环全景（一个 global_step 的五个阶段）

| 线索 / 问题 | 要点 / 解答 |
|------------|------------|
| 一个 step 做了啥？ | ①生成 → ②打分 → ③算 log_prob → ④算 advantage → ⑤更新参数。外层 `while` 一圈 = 一个 `global_step`。 |
| 阶段1 生成 | `prepare_rollout_engine()` 同步 actor 权重给 vLLM → vLLM 生成回答 → `release_rollout_engine()` 释放显存。 |
| 阶段2 打分 | `reward_fn.compute_reward()` 读**已生成文本**算分。纯解析/比对，**不跑模型 forward**。 |
| 阶段3 log_prob | 用 FSDP actor forward **精确重算** 生成文本的 log_prob（vLLM 不输出可靠 log_prob）。 |
| 阶段4 advantage | 把 reward → 组内归一化 → advantage（driver/CPU 上算，很轻）。 |
| 阶段5 更新 | RPC 调 `update_actor`，真正的 loss 计算在 **worker 内部**，不在 `fit()` 里。 |

## 二、打分 vs advantage（易混点 ★）

| 线索 / 问题 | 要点 / 解答 |
|------------|------------|
| 打分 = 算 advantage 吗？ | **不是，两步。** |
| 打分（阶段2） | 绝对分。这条回答好不好。规则算（IoU、答案对错）。各 prompt 不可比。 |
| advantage（阶段4） | 相对优势。比同组其他回答好多少。组内归一化 `(r-均值)/std`。可直接进 loss。 |
| 为啥不能直接拿 reward 当信号？ | 简单题都高分、难题都低分 → 直接用会误导模型偏好简单题。归一化后与难度脱钩，只看「组内谁更好」。 |
| ❌ 我的错误理解 | 以为打分就是算 advantage。**纠正**：打分是原料，advantage 是成品。 |

---

## 三、log_prob 家族（约束策略用，不是算 reward）

| 名称 | 来源 | 用途 |
|------|------|------|
| `old_log_probs` | actor forward(原图)，冻结 | PPO clip 比值的分母基准 |
| `log_probs`(新) | actor forward(更新中，带梯度) | 比值分子，被优化对象 |
| `aug_log_probs` | actor forward(遮掩图) | PAPO 感知 KL |
| `ref_log_probs` | reference 模型 | 防偏离基座的 KL |

| 线索 / 问题 | 要点 / 解答 |
|------------|------------|
| old 和 new 同模型为啥不同？ | 差在**时间**：old = update 前冻结；new = update 中反复算，策略在动。 |
| 为啥 vLLM 的 log_prob 没用？ | vLLM 为速度做近似，log_prob 不精确；且常不返回。必须用 FSDP actor 重算。 |
| 此刻 actor 与 vLLM 同策略吗？ | 同。阶段1 已同步权重，所以「重新推理一遍结果应一样」——理解正确。 |

## 四、PPO Clip Loss 理论（核心损失）

```
ratio = exp(log_probs - old_log_probs) = π_new(a|s) / π_old(a|s)
L = min( ratio·A,  clip(ratio, 1-ε_low, 1+ε_high)·A )
```

| 线索 / 问题 | 要点 / 解答 |
|------------|------------|
| clip loss 是组内优势吗？ | **不是。** advantage(A) 是 clip loss 的输入。先算 A，再喂进 clip loss。 |
| clip 干啥？ | 限制单步策略更新幅度，防训练崩。A>0 升概率但有上限，A<0 降概率有下限。 |
| 总损失 | `PPO_clip + kl_coef·KL(π‖π_ref) + kl_prcp_coef·KL(原图‖遮掩图) - entropy_coef·entropy` |

## 五、token_level_rewards（易混点 ★）

| 线索 / 问题 | 要点 / 解答 |
|------------|------------|
| 形状为啥是 (batch, 回答长度)？ | PPO 框架要求逐 token 信号。但 outcome reward 只有一个总分。 |
| 怎么塞？为啥放最后一个 token？ | 总分放**最后一个有效 token(EOS)**，前面填 0。对应「延迟奖励」——走到终点才结算。 |
| `token_level_scores` vs `token_level_rewards` | scores = 打分原始结果；rewards = scores 扣掉 KL 惩罚（`disable_kl=true` 时两者相等）。 |
| ❌ 困惑点 | 「为啥放最后/为啥适应」= 把单个总分套进 PPO 的逐 token 张量格式，纯格式适配，不改变「整句一个分」。 |

## 六、Critic（GRPO 为啥不用）

| 线索 / 问题 | 要点 / 解答 |
|------------|------------|
| critic 是啥？ | 价值网络 V(s)，传统 PPO 第二个模型，估「当前状态期望 reward」做 baseline：`A = reward - V(s)`。 |
| GRPO 为啥不用？ | 用**组内平均**代替 critic 做 baseline：`A[i]=(r[i]-组内均值)/std`。 |
| 好处 / 代价 | 省一整个模型（显存+计算砍半，不用训难训的 critic）；代价是每 prompt 要生成多个回答(n≥2)。 |
| ✅ 我的理解 | 「critic 是 PPO 第二个更优模型，GRPO 用组内相对优势代替」—— 正确。小修正：GAE 是用 critic 算 advantage 的**方法**，critic 是模型。 |

---

## 七、辅助 SFT 损失（use_sft_loss）

| 线索 / 问题 | 要点 / 解答 |
|------------|------------|
| 是啥？ | 额外加的一项 loss，和 PPO loss 相加（coef=1e-3，很轻）。不是替代。 |
| 流程 | 生成 → 打分 → `accuracy>0.1` 置 1 做 `correctness_mult_mask` → 只对答对样本算 SFT loss。 |
| 公式 | `sft_loss = masked_mean(-log_probs, combined_mask)` —— NLL，拉高正确回答 token 概率。 |
| ❌ 我的错误理解 | 以为 SFT 是「比较输入输出差异」。**纠正**：SFT 没有对比，就是 teacher-forcing NLL，给定前缀拉高目标 token 概率。 |
| 传统 SFT vs 自我 SFT | **公式完全一样**，唯一区别：target token 来源。传统=人工标注；自我=模型自产且答对的回答。 |
| 「变牢」机制 | 把已做对回答的生成概率(如 0.5)进一步推高(→0.7)，固化正确行为，防 RL 中被冲掉（防遗忘）。 |
| vs PPO | PPO 对差回答会降概率(A<0)；自我 SFT 只升不降（只动答对的），单向巩固。 |

## 八、SFT vs RL 宏观对比

| 维度 | SFT | RL(PPO/GRPO) |
|------|-----|--------------|
| 本质 | 模仿（学正确答案） | 试错+排序（学好坏差异） |
| 信号 | 绝对、高密度、稳 | 相对、低密度、方差大 |
| 探索 | 无 | 有 |
| 负反馈 | 无 | 有（压低错的） |
| 数据 | 需人工标注 | 只需 reward 函数 |
| 上限 | = 标注质量 | 可超越示范 |
| 成本/稳定性 | 低 / 稳 | 高 / 难调 |

| 线索 / 问题 | 要点 / 解答 |
|------------|------------|
| 标准管线 | 预训练 → SFT → RL。先 SFT 拉到「能用」，RL 才有起点（否则冷启动 reward 全 0 学不动）。 |
| SFT 适合 | 有高质量标注、教新能力/格式、训练早期、唯一标准答案。 |
| RL 适合 | 有自动 reward（数学/代码/IoU/偏好）、想超越标注、开放任务、模型已有基础。 |
| 本项目位置 | RL 阶段（GRPO/DAPO），假设 Qwen2.5-VL 已 SFT。`use_sft_loss` = RL 里掺点 SFT 锚定稳定。 |

## 九、参数更新 & 权重同步

| 线索 / 问题 | 要点 / 解答 |
|------------|------------|
| 全量还是部分更新？ | **全量**。本项目无 LoRA（只有 `freeze_vision_tower:false`，视觉编码器也训）。 |
| advantage 怎么驱动更新？ | A → PPO clip loss → `loss.backward()` 算梯度 → AdamW 更新 → FSDP 跨卡 all-reduce 同步。 |
| `critic_warmup` 是啥？ | critic 预热步数门槛。GRPO 下=0，恒成立，等于没启用。不是「有 critic 才更新」。 |
| `reduce_metrics(...)` | 聚合多卡/多 mini-batch 指标求平均，纯日志，不影响训练。 |
| actor 更新后怎么传给 vLLM？ | `fsdp_vllm.py:_sync_weight_to_vllm`：`get_model_state_dict` 聚合 FSDP 分片 → `model.load_weights` 灌进 vLLM。**显存内拷贝，不落盘。** |
| 为啥要同步？ | PPO 是 on-policy，必须用当前策略采样。actor 变了 vLLM 必须拿新权重。 |
| sleep/wake | vLLM 训练时 sleep 释放显存、生成时 wake_up；对应 `prepare`/`release_rollout_engine`。 |

## 十、on-policy & 采样多样性

| 线索 / 问题 | 要点 / 解答 |
|------------|------------|
| on-policy 啥意思？ | 用当前策略采的数据更新当前策略，更新完→新参数→重采→再更新，往复。 |
| off-policy 对立面 | 用旧策略/别人采的数据（如 DQN 经验回放）。PPO 是 on-policy，不能复用旧 batch。 |
| 同 prompt 怎么生成不同回答？ | **不是不同 seed，是概率采样。** 模型每步输出概率分布，按概率随机抽 n 次。 |
| temperature=1.0 | 控制分布尖锐度。T→0 趋贪心(全相同)，T=1 充分探索。GRPO 必须 T>0，否则组内全同 std=0 白训。 |
| top_p=0.99 | nucleus sampling，只从累计概率前 99% token 抽，砍长尾垃圾。 |
| seed=1 作用 | 只保证整体实验复现，不是每条回答配一个 seed。 |

## 名词对照（ML 通用 → 本框架）

| ML 通用 | 本框架 | 含义 |
|---------|--------|------|
| epoch | `total_epochs` (15) | 遍历完整训练集次数 |
| iteration/step | `global_step` | 一次完整 rollout+更新循环（比传统 step 重得多） |
| 最大步数 | `max_steps` (null) | 步数上限，与 epochs 取先到 |
| gradient step | mini-batch + `gradient_accumulation` | actor 内一次 backward |

`training_steps = num_examples // rollout_batch_size * total_epochs`

## 总结（Summary）

> **一句话串全程**：vLLM 按概率采样为每个 prompt 生成 n 个回答(on-policy) → reward_fn 打绝对分(放末位 token) → 扣 KL 得 token_level_rewards → 组内归一化成相对 advantage → 喂进 PPO clip loss（+可选 ref KL / PAPO 感知 KL / 辅助 SFT / entropy）→ FSDP 全量反向更新 actor → 显存内同步权重给 vLLM → 下一轮。
>
> **GRPO 的精髓**：用「组内多样采样 + 相对排序」替代 critic，省一个模型。所以 temperature>0 和 n≥2 是命根子。
>
> **三个关键纠错点**：①打分≠advantage（原料 vs 成品）②SFT 不是对比输入输出，是 teacher-forcing NLL ③critic_warmup 不是更新前提，GRPO 下形同虚设。
